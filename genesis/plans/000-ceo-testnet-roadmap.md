# robopoker — CEO Testnet Roadmap (CEO-SIGNED 2026-06-03)
Testnet north star: A public, reproducible NLHE benchmark where a trained robopoker blueprint bot beats a named baseline head-to-head, with every match downloadable as a replayable, signed transcript.
signed_off: true  (CEO sign-off 2026-06-03)

## Goals (sequenced now -> testnet)
- [medium] Produce a real, reproducible blueprint artifact — the GTO bot has to exist before it can compete.
  - proof: `trainer --cluster` then `trainer --fast` produces a non-empty blueprint in Postgres on a documented small-abstraction config; `trainer --status` reports trained epochs + row counts.
  - proof: `DatabasePlayer::from_database` hydrates that blueprint and plays a full hand vs `Fish` without panicking (exercises the just-landed `Flagship::hydrate` end-to-end).
- [medium] Make hand histories durable and replayable — a benchmark is only credible if every hand can be re-derived.
  - proof: the `unimplemented!()` `Schema` contracts in `records/{hand,play,participant}.rs` are implemented; a played hand round-trips DB write -> `Replay` read -> identical action sequence.
  - proof: a committed integration test plays N hands through `Room` and asserts persisted transcripts replay deterministically.
- [medium] Stronger named baseline shipped (STW-012): the v3 `PreflopBot` (preflop-tier aware rule bot) is the next-iteration target after the v2 `EquityBot`. A v4 stronger baseline (e.g. a second trained config) is the next slice if a v4 is needed.
- [medium] Stronger named baseline shipped (STW-013): the v4 `BlufferBot` (semi-bluff-aware rule bot — reuses the v3 preflop tier table + adds a postflop semi-bluff raise on checked-to boards with weak hands and no draw) is the next-iteration target after the v3 `PreflopBot`. The v4 is a *named rule-based* baseline, not a second trained config (which would be a much larger slice — a full retrain + a new `NlheProfile` variant + a new `DatabasePlayer` path). **STW-017 is landed**: the v5 "second trained config" slice the BlufferBot section defers to is in. The v5 trained config is `Flagship2` = `DiscountedRegret` + `QuadraticWeight` + `PluribusSampling` (vs the v4 `Flagship` = `PluribusRegret` + `LinearWeight` + `PluribusSampling`). The v5 has its own persistence pair (`blueprint_v2` + `epoch_v2` / `'current_v2'` + `staging_v2`), its own trainer (`trainer --fast2` running `Fast2Session`), its own bench seat (`RBP_BENCH_BLUEPRINT=v2` seats a `DatabasePlayer2` at seat 0 and reports `blueprint:"v2"` in the JSON), and its own status read (`trainer --status` now prints both v1 and v2 epoch + blueprint row counts). A v6 (e.g. a third DCFR-with-LinearWeight variant, or a "named bot vs second trained config" comparison) is the next slice if the v5 trained config proves meaningfully different from the v4. **STW-018 is landed**: the v6 comparison half the BlufferBot section defers to is in. A new `Mode::Compare` arm wires `trainer --compare` into the binary; `bench::run_compare` seats the v1 `DatabasePlayer` (seat 0) vs the v2 `DatabasePlayer2` (seat 1) in the same `Room` over K heads-up hands, reads the per-hand settlements for both seats, and prints a single-line JSON `CompareReport` declaring the winner (`"v1"`, `"v2"`, or `"tie"`) with the per-side mbb/100 / CI / win-rate numbers and the v1-vs-v2 `delta_mbb_per_100`. The compare reuses the existing `Room::play_hand_once` + `Room::settlements` pair the bench uses, gated by `RBP_COMPARE_HANDS` (default 200) and `RBP_COMPARE_BLIND` (default `B_BLIND`), and is covered by 6 new lib tests + 1 new `compare_run_emits_parseable_json_with_consistent_accounting` integration test that drives a real `trainer --reset` then `trainer --compare` end-to-end through a subprocess.
- [long] Public reproducible benchmark surface — the externally-verifiable testnet milestone.
  - proof: a `bench` harness runs blueprint-bot vs a named baseline over K hands (mirrored duplicate dealing), emits mbb/100 with a confidence interval, and writes a transcript bundle anyone can replay from the README. **STW-014** is landed: the in-memory `Transcript` bundle type is in, the bench harness writes one `transcript-<hand_id>.json` per hand under `RBP_BENCH_TRANSCRIPT_DIR`, and an integration test proves a real `Room` write → `Transcript::new` → `verify()` → `to_json()` → `serde_json::from_str` round-trip. **STW-015** is landed: three `Transcript` methods (`read_from_path`, `rebuild_action_sequence`, `replay_to_path`) a downstream consumer can call to re-derive the `(Position, Action)` sequence + a renderable text summary from a `transcript-<id>.json` produced by the bench. A live-Postgres end-to-end test (`transcript_replay_end_to_end`) drives a real `Room` write, reads the persisted records back, writes the transcript to a temp file, and proves the new API round-trips the action sequence and renders the expected seat/action text. **STW-016** is landed: `trainer --replay <path>` (and the lone-positional shortcut `trainer <path>`) wire the STW-015 public surface into the `trainer` binary. The new mode is read-only (no DB connection, no schema), prints the rendered transcript to stdout on success, prints a one-line diagnostic to stderr and exits 2 on a missing/corrupt/missing-arg file, and is covered by three pure-in-memory lib tests (`replay_run_renders_fixture_transcript`, `replay_run_errors_on_missing_file`, `replay_run_errors_on_corrupt_file`) plus a hand-written CLI smoke against a fixture `transcript-*.json`.
  - proof: harden auth (STW-004) so the WebSocket play surface can be exposed; a remote client connects via `/room/enter` and plays the hosted bot.
  - proof (operator-visible): a documented shell-level runbook an operator (or a cron worker) can run against a real Postgres to produce a single receipt directory. **STW-019** is landed: `scripts/testnet-live-proof.sh` is a pure-bash driver that runs the full chain (`--cluster` → `--reset` → `--smoke` → `--status` → `--bench` → `--compare` → `--replay <transcript>`) as subprocesses against a single Postgres, captures each step's stdout + stderr + exit code into a per-step `receipts/testnet-live-proof-<UTC-ISO>/<step>/{stdout,stderr,exit}.txt` layout, and emits a one-line `testnet live_proof complete: smoke=N status=N bench=N compare=N replay=BYTES` headline in `SUMMARY.txt` a dashboard can grep. `DATABASE_URL` is forwarded as `DB_URL`; the script refuses to run with exit 3 when neither is set. **STW-023** is landed: the runbook now writes a `recipe.json` machine-readable manifest alongside `SUMMARY.txt` (the JSON shape mirrors the `crates/autotrain::LiveProofRecipe` struct: `trainer_bin`, `database_url` (redacted), `steps: [{ name, exit, stdout_bytes, stderr_bytes }, ...]` in the pinned `STW023_CHAIN_STEPS` order), and a new no-DB integration test `cargo test -p rbp-autotrain --test live_proof_receipt` drops a synthetic receipt under `target/test-receipts/live_proof-fixture-<UTC>/`, calls the shared `LiveProofReceipt::verify` verifier on it, and asserts the verifier's green verdict + the `recipe.json` step-name order matches the on-disk `receipts/<step>/` directory layout. A regression in either surface (renamed step, dropped exit code, broken headline prefix, missing `recipe.json`) fails the shared verifier — the operator-visible receipt and the CI-visible receipt now share one source of truth. The `cargo test -p rbp-autotrain --features database --test live_proof` integration test (which chains the same seven steps in a single `cargo test` invocation) and the no-DB `cargo test -p rbp-autotrain --test script_shape` shell-shape pinner (which asserts the runbook script exists + is executable + parses with `bash -n`, the runbook doc lists every env knob and every chain step, and the runbook script sources the new `recipe.json` block) close the gap on both sides: the per-step contracts in CI, the per-step operator receipts in production.
  - proof: the full `cargo test --workspace -- --test-threads=4` chain is stable across 3 consecutive runs. **STW-020** is landed: `scripts/workspace-parallel-proof.sh` is a pure-bash driver that runs the full workspace test 3 times back-to-back against the current source tree, captures each run's stdout + stderr + exit code into a per-run `logs/workspace-parallel-proof/<UTC-ISO>/run-{1,2,3}/{stdout,stderr,exit}.txt` layout, and emits a one-line `workspace parallel proof complete: runs=3 failures=0` headline in `SUMMARY.txt` (exit 3 on any run failure, exit 1 on script-internal error). Knobs: `RBP_WORKSPACE_PARALLEL_THREADS` (default 4), `RBP_WORKSPACE_PARALLEL_RUNS` (default 3), `RBP_WORKSPACE_PARALLEL_SKIP_BUILD` (set to 1 to skip the one-time pre-build). The script's inner `cargo test --workspace` filters out the recursive test driver via `--skip=runbook_run_exits_zero_with_single_clean_workspace_run` to prevent infinite recursion. The `cargo test -p rbp-autotrain --test workspace_parallel_proof` integration test pins the runbook's surface (script exists + is executable + parses with `bash -n`, SUMMARY.txt headline format drift) and drives a single-run proof end-to-end via `RBP_WORKSPACE_PARALLEL_RUNS=1` so `cargo test --workspace` stays green. The `bust_prevents_next_deterministic` lib test in `crates/gameplay/src/game.rs` threads a seeded `StdRng` through a new `Deck::deal_with` injection seam and asserts the full pot conservation invariant (sum=200, len=2, each ∈ {0, 100, 200}, sorted ∈ {{0, 200}, {100, 100}}) across 64 fixed seeds — making the heads-up all-in showdown property bit-exactly reproducible across machines and CI runners. The `verification:workspace-parallel` hinge is closed.

## Lens verdicts
CEO: Testnet is the right north star but for an engine repo it means a public reproducible benchmark, not a chain deploy. robopoker's highest ecosystem leverage is being the proof that "trained strategy is real and replayable" — the same claim myosu and poker-arena depend on. The blueprint-exists + transcript-replays + beats-baseline trio de-risks the whole game-solving thesis; prioritize it over the live WebSocket casino, which is downstream.
Design: A credible engine testnet demo MUST show artifacts a stranger can re-run: a trained blueprint, a head-to-head result with a confidence interval, and downloadable transcripts that replay bit-for-bit. Today the only opponent is `Fish` (random) and hand histories panic on persist, so there is nothing to show or replay. The receipts gap, not the UI, is the blocker.
Eng: The hardest blocker (`Flagship::hydrate`, STW-003) just landed, unblocking the DB-backed bot path. The remaining critical path is concrete and bounded: implement the panicking `Schema` record contracts, add a reproducible small-scale train, and write the benchmark harness. No new chain or transport integration is required for the engine testnet milestone — it is all within the existing crates. **STW-021** ships the TUI headless QA report as a real gate (per-check `passed` / `detail` / `id` breakdown in `tui.qa.json` + a `## QA Checks` section in `tui.receipt.md` a testnet dashboard can `grep '^QA-CHECK tui\.'`); the previously-static `assertions: Vec<&'static str>` is now a `Vec<&'static str>` of failing check ids (empty for green runs) so the existing receipt shape is backward-compatible. **STW-022** ships the plan-vs-reality staleness gate: `scripts/plan-staleness-gate.sh` (pure bash) + `crates/autotrain/tests/plan_staleness.rs` (no-DB pinner) mechanically detect any `[ ] [P0]` row in the CEO roadmap that duplicates a `- [x] \`STW-NNN\`` row in `IMPLEMENTATION_PLAN.md`, exit 3 on a ghost, and exit 0 with `plan staleness gate complete: checked=N ghosts=0` on a clean surface — so the P0 retirement can't be silently un-retired by a future roadmap edit. **STW-023** lands the shared `LiveProofReceipt` verifier (`crates/autotrain/src/receipt.rs` + 7 lib tests) plus a `recipe.json` machine-readable manifest the `scripts/testnet-live-proof.sh` runbook writes alongside `SUMMARY.txt` (mirror of the `LiveProofRecipe` struct); a new `crates/autotrain/tests/live_proof_receipt.rs` (no-DB, 5 tests) drops a synthetic receipt, calls `LiveProofReceipt::read_and_verify`, and asserts the verifier agrees the receipt is green. The `crates/autotrain/tests/live_proof.rs` integration test now drops + re-verifies the same receipt shape at the end of the chain, so the operator-visible receipt and the CI-visible receipt share one verifier — closing the `testnet-live-proof` operator-receipt hinge.

## Sequencing & risks
The lenses agree the milestone is a reproducible benchmark with replayable transcripts, and disagree only on emphasis: Design wants receipts first, CEO wants the strategic proof trio, Eng wants the compile/persist path solid. These reconcile into one ordering — persistence (records Schema) and reproducible training first, because the benchmark and transcripts depend on both; auth hardening runs in parallel since it gates only the optional live WebSocket surface and has no dependency on training. Key risks: (1) full Pluribus-scale abstractions are 3GB+ and slow, so the P0 train MUST be a small env-gated config or the benchmark never runs in CI/worker time — "reproducible small bot beats random" is the honest first milestone, with full-scale parity a later goal. (2) "named baseline" is currently only `Fish`; beating random is necessary but weak, so frame v1 as "blueprint beats random with significance" and treat a stronger named baseline (e.g. a rule-based or a second trained config) as the next slice — **STW-012 ships the v2 (`EquityBot`) and v3 (`PreflopBot`) rule-based baselines**, so the v1 framing is now demonstrably weaker than the bench's `RBP_BENCH_BASELINE=preflop` runs prove; a v4 (a second trained config) is the next slice if the v3 baseline proves insufficient. (3) hand-history determinism depends on capturing the dealt deck/seed in `Replay`; the round-trip test must pin that or transcripts won't replay.
